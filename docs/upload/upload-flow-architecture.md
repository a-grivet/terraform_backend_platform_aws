# Upload Flow Architecture

## Introduction

This document explains the upload flow in this repository.

The goal of this code is to provide a secure upload foundation for Terraform template ZIP files:

1. the client asks the backend to prepare an upload,
2. the backend creates an upload intent in DynamoDB and returns a presigned S3 URL,
3. the client uploads the ZIP directly to S3,
4. the client confirms the upload,
5. the backend verifies the object in S3 and marks the upload as completed.

The implementation currently covers the upload foundation only. It does not yet include later steps such as archive cleanup, validation workflow orchestration, or deployment execution.

## Sources of Truth

The main source-of-truth directories for this upload foundation are:

- `terraform/upload-foundation/` for infrastructure and deployment configuration
- `lambdas/` for the upload Lambda handlers and their colocated unit tests
- `docs/upload-flow/` for design notes, contracts, implementation notes, and handover documentation

## What This Code Provides

### Deployed Services Index

| Service | Terraform / code location | Role | Key features |
| --- | --- | --- | --- |
| S3 raw upload bucket | `terraform/upload-foundation/main.tf` | Stores uploaded ZIP files | Private bucket, public access blocked, server-side encryption, configurable versioning, CORS for direct browser `PUT` |
| DynamoDB upload intents table | `terraform/upload-foundation/main.tf` | Stores upload metadata and status | Composite key (`template_id`, `version_id`), tracks `PENDING` and `UPLOADED`, stores expected size, actual size, object location, timestamps, and `etag` |
| API Gateway HTTP API | `terraform/upload-foundation/main.tf` | Exposes the upload endpoints | Same API for both routes, auto-deployed stage, explicit throttling to avoid `429` misconfiguration |
| `prepare-template-upload` Lambda | `lambdas/prepare-upload/handler.py` | Creates an upload intent and presigned URL | Validates request payload, generates opaque IDs, stores `PENDING` item, creates regional S3 presigned `PUT` URL |
| `complete-template-upload` Lambda | `lambdas/complete-upload/handler.py` | Confirms an uploaded ZIP | Loads the DynamoDB item, checks `PENDING`, calls `HeadObject`, verifies size, updates status to `UPLOADED` |
| IAM roles and policies | `terraform/upload-foundation/main.tf` | Grants runtime permissions to Lambdas | CloudWatch Logs, DynamoDB access, S3 object access, optional KMS permissions |
| CloudWatch log groups | `terraform/upload-foundation/main.tf` | Stores Lambda execution logs | Separate log groups for both Lambdas, 14-day retention |

### Exposed API Endpoints

| Method and route | Backing Lambda | Purpose |
| --- | --- | --- |
| `POST /templates/uploads/prepare` | `prepare-template-upload` | Creates the upload intent and returns the presigned S3 URL |
| `POST /templates/uploads/complete` | `complete-template-upload` | Verifies the uploaded object and marks the intent as `UPLOADED` |

### Runtime Behavior Summary

- Uploads are stored in S3 under `uploads/raw/<template_id>/<version_id>.zip`.
- `template_id` and `version_id` are generated by the backend as opaque identifiers.
- `template_name` and `file_name` are provided by the frontend or API client.
- Upload status currently follows this path:
  - `PENDING` after `POST /templates/uploads/prepare`
  - `UPLOADED` only after `POST /templates/uploads/complete` verifies the object in S3 successfully

### Main Functional Features

- Direct S3 upload from the client by presigned URL
- Backend-owned upload state in DynamoDB
- Server-side verification before status transition to `UPLOADED`
- Regional S3 presigned URLs to avoid redirect and token issues
- Configurable bucket naming, region, throttling, size limit, encryption, and CORS

### Configuration And Infrastructure Features

This section summarizes the most important infrastructure configuration points for the three core technical building blocks behind the upload flow.

#### S3 Raw Upload Bucket

The raw upload bucket is the storage entry point for uploaded ZIP files.

Current configuration and features:

- bucket name is configurable through `upload_bucket_name`
- if not explicitly set, the default naming pattern is `inca-terraform-uploads-<environment>-<account_id>`
- the bucket is private by default
- public access is explicitly blocked through S3 Public Access Block
- server-side encryption is always enabled
- encryption can use:
  - `AES256` by default
  - `aws:kms` when `encrypt_with_kms = true`
- bucket versioning is configurable through `enable_versioning`
- bucket CORS is configured for direct browser or client `PUT`
- allowed CORS origins are configurable through `allowed_cors_origins`
- the upload object path is controlled by the backend under:
  - `uploads/raw/<template_id>/<version_id>.zip`

Operationally, this means the frontend never decides the bucket name or object key. It only receives a presigned URL and uploads the ZIP directly to the location determined by the backend.

#### DynamoDB Upload Intents Table

The DynamoDB table is the source of truth for upload state and metadata.

Current configuration and features:

- table name is configurable through `dynamodb_table_name`
- billing mode is `PAY_PER_REQUEST`
- the primary key is composite:
  - partition key: `template_id`
  - sort key: `version_id`
- server-side encryption is enabled
- when KMS encryption is enabled for the stack, the table also uses the provided KMS key
- the table stores:
  - business metadata such as `template_name` and `file_name`
  - expected upload metadata such as `content_type` and `expected_size_bytes`
  - storage location metadata such as `s3_bucket` and `s3_key`
  - lifecycle metadata such as `created_at`, `updated_at`, `uploaded_at`
  - status transition data such as `PENDING`, `UPLOADED`, `actual_size_bytes`, and `etag`

In practice, the table protects the workflow from trusting the client for final storage metadata or upload status.

#### IAM Roles, Policies, And Permissions

The upload foundation creates one IAM role and one IAM policy per Lambda:

- `prepare-template-upload` role and policy
- `complete-template-upload` role and policy

Current configuration and features:

- both roles are assumed by the Lambda service through `sts:AssumeRole`
- both roles receive CloudWatch Logs permissions for runtime logging
- the current Terraform stack attaches the same policy document structure to both Lambda roles
- the policy grants DynamoDB access on the upload intents table
- the policy grants object-level S3 access on the raw upload bucket
- optional KMS permissions are added when `encrypt_with_kms = true`

The currently granted permissions are:

- CloudWatch Logs:
  - `logs:CreateLogGroup`
  - `logs:CreateLogStream`
  - `logs:PutLogEvents`
- DynamoDB:
  - `dynamodb:GetItem`
  - `dynamodb:PutItem`
  - `dynamodb:UpdateItem`
  - `dynamodb:Query`
- S3 object access:
  - `s3:GetObject`
  - `s3:HeadObject`
  - `s3:PutObject`
- Optional KMS access:
  - `kms:Decrypt`
  - `kms:Encrypt`
  - `kms:GenerateDataKey`
  - `kms:DescribeKey`

From a runtime perspective:

- `prepare-template-upload` uses these permissions to create the upload intent and generate the presigned upload contract
- `complete-template-upload` uses these permissions to read the upload intent, verify the object in S3, and update the final upload status

### How The Upload Flow Is Used

The expected client sequence is:

1. `POST /templates/uploads/prepare`
2. Receive:
   - `template_id`
   - `version_id`
   - `status = PENDING`
   - `upload.url`
3. Upload the ZIP directly to S3 with the presigned `PUT` URL
4. `POST /templates/uploads/complete`
5. Receive:
   - `status = UPLOADED`

### What The Frontend Must Send

For `POST /templates/uploads/prepare`, the client must provide:

- `template_name`: business name chosen in the UI
- `file_name`: original ZIP filename from the user machine
- `content_type`: ZIP MIME type
- `file_size`: ZIP size in bytes

For `POST /templates/uploads/complete`, the client must provide:

- `template_id`
- `version_id`

The client must not decide or send:

- S3 bucket name
- S3 object key
- final status
- actual object size

Those values are controlled by the backend.

### 4. End-To-End Manual Test Through API Gateway

This is the main real-world test of the upload flow.

First prepare the environment:

```bash
cd terraform/upload-foundation
API_URL="$(terraform output -raw upload_http_api_endpoint)"
FILE_SIZE="$(stat -c%s /tmp/inca-upload-network-lab.zip)"
```

#### Step A: Prepare Upload

```bash
curl -s -X POST "${API_URL}/templates/uploads/prepare" \
  -H "Content-Type: application/json" \
  -d "{
    \"template_name\": \"upload-network-lab-status-transition\",
    \"file_name\": \"inca-upload-network-lab.zip\",
    \"content_type\": \"application/zip\",
    \"file_size\": ${FILE_SIZE}
  }" > /tmp/status-transition-prepare.json

cat /tmp/status-transition-prepare.json
```

Expected result:

- JSON response with `template_id`
- JSON response with `version_id`
- `status = PENDING`
- a presigned S3 `upload.url`

Extract the values for the next steps:

```bash
eval "$(python3 - <<'PY'
import json
with open('/tmp/status-transition-prepare.json') as f:
    body = json.load(f)
print("TEMPLATE_ID='{}'".format(body["template_id"]))
print("VERSION_ID='{}'".format(body["version_id"]))
print("UPLOAD_URL='{}'".format(body["upload"]["url"]))
print("S3_KEY='{}'".format(body["object"]["key"]))
PY
)"
```

#### Step B: Upload The ZIP To S3

```bash
curl -v "$UPLOAD_URL" \
  -H 'Content-Type: application/zip' \
  --upload-file /tmp/inca-upload-network-lab.zip
```

Expected result:

- `HTTP/1.1 200 OK`

Optional verification:

```bash
aws s3api head-object \
  --bucket "$(terraform output -raw upload_bucket_name)" \
  --key "$S3_KEY"
```

#### Step C: Confirm The Upload

```bash
curl -s -X POST "${API_URL}/templates/uploads/complete" \
  -H "Content-Type: application/json" \
  -d "{
    \"template_id\": \"${TEMPLATE_ID}\",
    \"version_id\": \"${VERSION_ID}\"
  }" > /tmp/status-transition-complete.json

cat /tmp/status-transition-complete.json
```

Expected result:

- JSON response with `status = UPLOADED`

#### Step D: Check DynamoDB

Open the upload intents table in AWS and verify that the same item moved:

- from `PENDING`
- to `UPLOADED`

You should also see fields such as:

- `uploaded_at`
- `actual_size_bytes`
- `etag`

## Security

### Infrastructure-Level Protection

- The S3 upload bucket is private.
- Public ACLs and public bucket policies are blocked.
- Server-side encryption is always enabled:
  - `AES256` by default
  - optional KMS if configured
- The DynamoDB table also uses server-side encryption.

### Direct Upload Security Model

- Files are not uploaded through the backend Lambda payload path.
- The backend returns a short-lived presigned S3 `PUT` URL.
- Presigned URLs are generated for the bucket region explicitly, which avoids redirect and invalid-token issues.
- The URL expiration is configurable through Terraform.

### Backend Validation

`prepare-template-upload` validates:

- required JSON body
- `template_name`
- `file_name`
- `.zip` extension
- accepted content types
- positive integer file size
- maximum allowed size

`complete-template-upload` validates:

- `template_id`
- `version_id`
- current status must still be `PENDING`
- uploaded object must exist in S3
- uploaded object size must match `expected_size_bytes`

Only after these checks does the backend move the item to `UPLOADED`.

### Access Control And Least Privilege

The Lambdas use dedicated IAM roles and policies. Their permissions are scoped to:

- CloudWatch Logs
- the upload intents DynamoDB table
- objects inside the upload bucket
- optional KMS key access when enabled

### API Protection

- API Gateway throttling is explicitly configured to avoid an accidental `0 / 0` stage configuration.
- The current stack does not yet include end-user authentication or an authorizer. This should be added before production exposure.

## Current Gaps And Hygiene Notes

The current upload foundation works, but these topics are still open for future improvement:

- cleanup of stale `PENDING` intents
- cleanup of orphaned S3 objects
- long-term retention policy for `UPLOADED` files
- authentication and authorization for API clients
- validation pipeline after `UPLOADED`
- more advanced monitoring and alerting

In other words, this stack is a solid upload MVP, but not yet the complete production lifecycle.
